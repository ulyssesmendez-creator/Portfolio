Build Your Own World

(Build Your Own World) - World Generation System

Overview

BYOW focuses on generating interactive, tile based worlds. The project emphasizes algorithmic design, deterministic randomness, and state management.

Problem

World generation for real world games requires balancing randomness with structure. These random worlds, like dungeon type games, must feel varied while remaining navigable. They must also be reproducible and logically consistent for the players. Managing player interaction, rendering, and game state adds complexity as these worlds grow in size.

Solution

I designed a system that generates a variety of unique worlds using seeded randomness, ensuring the same input seed always produces the same world. The world consists of rooms, hallways, paths, keys, and walls allowing players to explore and interact with the environment in real time.

World Generation

The world generation logic constructs rooms and connects them with hallways without overlapping pathways and unreachable areas through the use of walls. Deterministic random seeds are used to maintain reproducibility, which is essential for debugging and testing.

System Design

BYOW is organized into components that separate world generation, player state, and input handling. The world is generated on a two dimensional tile grid using deterministic randomness using a unique seed, ensuring the same seed always produces the same layout. Rooms, hallways, and walkable tiles are generated first, after which the avatar is placed on a valid floor tile. Player movement updates the avatar‚Äôs position only when the destination tile is reachable, keeping world state and rendering consistent. Keyboard input is handled independently, allowing the same system to support both live input and scripted commands for testing.

What I Learned

This project strengthened my knowledge and understandability about large systems involving randomness, state, and interaction. It also reinforced the importance of reproducibility and modular design.

NGordnet

NGordnet ‚Äî Text Data Analysis System

Overview

NGordnet is a data driven text analysis system inspired by Google‚Äôs Ngram Viewer. The project focuses on exploring word usage trends over time by processing and querying large textual datasets.

Problem

Analyzing word trends across decades requires working with large datasets while supporting fast and flexible queries. Naive implementations struggle with performance and scalability when repeatedly comparing complex word frequencies over time.

Solution

NGordnet organizes the mess that comes with textual data into efficient data structures that allow users to query the usage of words over time, compare multiple terms, and analyze different trends.

System Design

The NGordnet backend separates data storage, processing, and querying. The word data is loaded once at startup and stored in maps that associate each word with a time ordered series of yearly counts, alongside a separate series tracking total word usage per year. A central API layer provides methods to retrieve counts, compute frequencies, combine multiple words, and restrict results to specific year ranges. Finally, user requests are translated to API calls that result in a visual graph for the user.

Data Structures & Algorithms

NGordnet uses hash maps and tree based structures to associate words with time series frequency data. These type of structures enable efficient range queries, aggregation, and comparison operations.

Applications & Use Cases

The system supports comparing word popularity over time, identifying historical word trends, and spotting out frequency changes across time. This system showcases real world workflows used in linguistic problems.

What I Learned

This project strengthened my understanding of system design and data structure selection. It also reinforced the importance of designing APIs around real search processing rather than only storing raw data.

ParkSpace

ParkSpace - Full Stack Location Based Parking Platform

Overview

ParkSpace is a full stack web application designed to address the problem of finding available parking spaces in dense urban environments. Instead of treating parking as a list of spaces, my platform models parking availability as a location aware resource that users can explore visually through an interactive map interface. I hope to set it apart from current parking app experiences.

Problem

Finding parking in cities is often time consuming and fragmented across different platforms. Many existing solutions lack real time context, intuitive visual search, or the flexibility for individuals to list private or informal parking spaces.

Solution

ParkSpace enables users to discover and list parking spaces based on proximity and real time location. Users can search for nearby parking, view available listings directly on a map, and interact with results that update as the map view changes.

Frontend Architecture

The frontend is built with React and Javascript using a component based architecture. The interface manages complex UI state, asynchronous data fetching, and map driven interactions.

Backend & Data

The backend uses Node.js and Express to expose RESTful APIs responsible for listing creation, validation, and querying. MongoDB is used to store parking listings with flexible schemas well suited for location based data. I also integrated firebase authentication to securely manage user accounts, handling signup, login, and session state.‚Äù

Future Work

ParkSpace was designed with scalability and extensibility in mind. Planned improvements include user authentication, availability scheduling, pricing logic, and analytics to better understand usage patterns and demand across different locations.

Scheme Interpreter

Scheme Interpreter ‚Äî Programming Language Implementation

Overview

This project involved building an interpreter for a subset of the Scheme programming language. The goal was to model how such a language evaluates expressions, manages variable scope, and executes function calls using recursion.

Problem

Interpreters must correctly handle expression evaluation, lexical scoping, and function calls. Small mistakes in evaluation order or environment handling can lead to subtle but difficult to debug errors. These topics are considered when programmers develop some new programming language.

Solution

I implemented a recursive interpreter that evaluates Scheme expressions according to the language syntax. The system parses expressions, resolves symbols through nested environments, and runs functions using a frame based environment model.

Evaluation Model

The interpreter uses recursive evaluation in which expressions are evaluated based on their type symbols, literals, procedure calls, or special forms. Each function call creates a new environment frame that connects it back to the original scope.

Environment & Scoping

Variable bindings are managed using chained environment frames, allowing the interpreter to resolve symbols correctly across nested scopes. This design enforces lexical scoping rules and ensures predictable behavior during function evaluation and recursion.

System Design

The interpreter is organized into components responsible for parsing, evaluation, and environment management. This separation of concerns makes the system easier to reason about, extend, and debug as additional language features are introduced.

What I Learned

This project deepened my understanding of how programming languages can work internally. In this case it showcases particularly how recursion, abstraction, and scoping are able to interact during execution. It also strengthened my ability to reason about complex systems where correctness depends on precise control and system management.
